# 租用 GPU 的分阶段计算协议

本协议规定“先便宜验证，后高规格确认”的执行方式。任何阶段未通过 Gate，都不得因为已经制定了大模型计划而继续扩大算力。

## 1. 选卡原则

1. 先按显存是否足够筛选，再比较单价与吞吐；
2. 24GB 足以完成 Qwen3-1.7B LoRA-DPO、批量生成和中小型 judge，不提前租 48/80GB；
3. 48GB 只用于 Qwen3-4B、多模型同时驻留或更大的本地 judge；
4. 80GB 只用于可回答新研究问题的 8B/更大确认；
5. 不以 GPU 型号替代实验设计质量。

## 2. R0 必测吞吐

首个 24GB 实例必须记录：

- 30-step 和 300-step LoRA-DPO 外推时间；
- 1,000 个 64-token 回答的生成时间；
- 1,000 个回答的每个安全 judge 评估时间；
- 1,000 个回答 embedding 时间；
- peak VRAM；
- 模型下载和环境安装时间单独记录，不混入 GPU 计算时间。

后续预算使用：

```text
预计卡时 = 训练 run 数 × 单 run 实测时间
         + 回答数量 / 实测每小时生成数
         + judge 数量 × 回答数量 / 实测每小时 judge 数
         + 20% 失败与重跑余量
```

## 3. 各档 GPU 的职责

### 24GB 低价档

候选：RTX 3090、RTX 4090，以及平台上同显存且兼容的软件栈。

负责：

- R0 环境 smoke；
- R1 公开 benchmark pilot；
- R2 数据轴单 seed；
- R3 Qwen3-1.7B 多 seed；
- R4 方法对照。

### 48GB 正式复核档

候选：L40S、RTX A6000。

负责：

- Qwen3-4B 2–3 seeds；
- 更大 batch 的 32/64-sample 生成；
- 量化或半精度的大型语义/安全 judge；
- 关键结果复核，不用于早期超参数搜索。

### 80GB 可选确认档

候选：A100 80GB；只有时间收益能抵消费用时选择 H100 80GB。

负责：

- 可选 8B 规模确认；
- 需要 80GB 才能回答的最终 judge 或消融；
- 不用于重复 24GB 已能完成的任务。

## 4. 实例生命周期

启动后：

1. clone 固定 commit；
2. 挂载持久化数据盘；
3. 运行 `python scripts/check_rental_environment.py`；
4. 先运行 dry/smoke；
5. 记录 run manifest；
6. 每完成一阶段立即同步 adapter、manifest 和聚合结果；
7. 确认同步完成后关机，避免空闲计费。

不得将 Hugging Face token、平台密钥或 SSH 私钥写入仓库。

## 5. 升级判断

- R0 失败：修环境，不继续计费跑实验；
- R1 失败：优先修正 judge/语义指标，不重新训练；
- R2 没有稳定条件差异：停止多 seed；
- R3 不跨 seed：不租 48GB；
- R5 方向反转：报告模型规模边界，不直接追加 80GB；
- 只有明确的新结论需要 8B 时，才进入 R6。
